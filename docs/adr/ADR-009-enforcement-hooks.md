# ADR-009: Hard Enforcement via PreToolUse Exit Code 2

**Status:** Accepted  
**Date:** 2026-06-30  
**Context:** Version 2 (CAP System Design v1)

## Context

Advisory instructions in CLAUDE.md achieve approximately 60% compliance for complex behavioral constraints (e.g., "delegate to orchestrator when editing 3+ files", "use worktrees for parallel writes"). The remaining 40% failure rate manifests as:

- Direct multi-file edits without delegation, leading to inconsistent changes
- Parallel agent writes to the same repo without worktree isolation, causing merge conflicts
- Repeated violations of the same rules across sessions despite corrections being recorded

**Key constraints:**
- Claude Code supports PreToolUse hooks that receive tool call JSON on stdin
- Exit code 2 from a PreToolUse hook physically blocks the tool call (hard enforcement)
- Exit code 0 allows the tool call to proceed
- Hooks must be fast (<100ms) to avoid UX degradation
- Enforcement state must persist across hook invocations (hooks are short-lived processes)

## Decision

**Use PreToolUse hooks with `exit(2)` and a stateful SQLite counter to enforce delegation rules.**

The enforcement hook (`src/cap/hooks/pretool.py`) runs before every tool call. For file-writing tools (`Edit`, `Write`, `NotebookEdit`), it tracks distinct files edited per session. When a non-delegated edit would touch a 3rd distinct file, the hook exits with code 2, blocking the operation and printing a clear error message explaining how to delegate or enable passthrough.

**Key design choices:**
- State tracked in `~/.cap/cap.db` (unified database) with `enforcement_edits` and `enforcement_violations` tables
- Agent context tracked via `agent_contexts` table — edits within an active Agent() context are marked as delegated and exempt
- Escape hatch: `cap passthrough` enables a temporary bypass (5-minute TTL, max 3/hour, fully logged)
- Worktree enforcement: multiple agents writing to the same repo must be in separate worktrees

## Alternatives Considered

| Alternative | Pros | Cons | Verdict |
|-------------|------|------|---------|
| **Advisory only (CLAUDE.md)** | Zero infrastructure, no false positives | ~60% compliance, repeated violations, no audit trail | Rejected (current state, insufficient) |
| **PostToolUse warning** | Non-blocking, informational | Does not prevent the bad action, only logs it after the fact | Rejected (too late) |
| **Custom MCP tool wrapper** | Could intercept at tool level | Claude Code does not support tool interception via MCP, only hooks | Rejected (not feasible) |
| **File system permissions** | OS-level enforcement | Too coarse, breaks legitimate single-file edits, no delegation awareness | Rejected |

## Consequences

### Positive
- **100% enforcement** for multi-file delegation rules on complex tasks
- **Stateful tracking** — violations are recorded with timestamps, tool names, and file paths for audit
- **Escape hatch** — `cap passthrough` provides a safe, logged, rate-limited bypass for legitimate exceptions
- **Fast execution** — SQLite WAL reads are <5ms; hooks complete well within 100ms budget
- **Agent-aware** — edits within an active Agent() delegation context are automatically exempt
- **Worktree safety** — parallel agent writes to the same repo are blocked unless in separate worktrees

### Negative
- **False positives** — legitimate 3+ file edits (e.g., a trivial rename across files) require either delegation or passthrough
- **Hook latency** — adds ~10ms per tool call for SQLite open + query (acceptable)
- **Complexity** — requires coordination between hook scripts and MCP servers (agent context registration)
- **Session ID dependency** — enforcement relies on Claude Code providing a consistent session_id in hook input

## Implementation Notes

**Hook registration** (generated by `cap init` into `.claude/settings.json`):
```json
{
  "hooks": {
    "PreToolUse": [{
      "command": "python3 ~/.cap/hooks/pretool.py",
      "timeout_ms": 500
    }]
  }
}
```

**Passthrough activation:**
```bash
cap passthrough              # 5-minute bypass
cap passthrough --ttl 120    # 2-minute bypass
cap passthrough --reason "trivial rename across 5 files"
```
